Security and Responsible Disclosure

Wallet safety, direct ENS verification, static-hosting limits, and reporting.

Never share seed phrases, private keys, recovery codes, or secrets. GoalOS support will never need them.

Access security

The access flow requests one MetaMask signature and then reads Ethereum Mainnet. Review the message carefully. The access flow should not request a transaction, gas payment, token approval, or asset transfer. Reject anything unexpected.

Direct-name verification

The release requires exact direct-child syntax and strict current ownership. It checks the ENS Registry for unwrapped names and configured Name Wrapper contracts for wrapped names, including expiry. Approvals, resolver records, and ENS text records do not activate membership.

Static-hosting boundary

GitHub Pages and public IPFS gateways distribute static bytes. Client-side gating can control the intended experience but cannot guarantee secrecy of files already published to a public host. Use separately reviewed encryption and decentralized key-release infrastructure when confidentiality is required.

Responsible disclosure

Do not publish vulnerabilities, wallet addresses linked to private matters, keys, or confidential evidence. Contact secretariat@montreal.ai privately with version, component, reproduction steps, and impact. Do not exploit beyond what is necessary to demonstrate the issue.

Deployment responsibility

Operators must perform independent dependency review, threat modelling, CSP review, RPC resilience, supply-chain controls, contract review where applicable, wallet testing, monitoring, incident response, backups, legal review, and controlled acceptance with owned test names before production.

No safe-harbour promise

This notice requests coordinated disclosure but does not create universal immunity, compensation, employment, agency, or a bug-bounty promise.
